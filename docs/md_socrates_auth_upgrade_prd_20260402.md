# Socrates 认证升级方案 PRD

> 日期: 2026-04-02
> 适用项目: `apps/socrates`
> 状态: 待实施
> 优先级: P0

---

## 一、目标

把当前 `手机号 + 密码` 的注册登录流程，逐步升级为更符合国内使用习惯的认证体系：

1. 先支持手机号验证码注册 / 登录
2. 再支持微信二维码快速登录 / 绑定
3. 最后把支付成功后的“补开户 / 快速进入系统”接进支付链路

一句话：

`先把手机号验证码做成主入口，再把微信接成高转化入口，最后再收口支付后的自动承接。`

---

## 二、当前现状

### 当前登录模型

当前并不是真正的手机 OTP 认证，而是：

- 输入手机号
- 把手机号映射成伪邮箱：`${phone}@student.local`
- 再走 Supabase `email/password`

当前关键文件：

- `apps/socrates/lib/contexts/AuthContext.tsx`
- `apps/socrates/app/(auth)/login/page.tsx`
- `apps/socrates/app/(auth)/register/RegisterPageV3.tsx`
- `apps/socrates/app/api/auth/register/route.ts`

### 当前支付模型

当前支付流是：

1. 用户先登录
2. 创建订单
3. 微信 / 支付宝支付
4. 支付回调写入订单与订阅
5. 成功页跳转系统

当前关键文件：

- `apps/socrates/app/(student)/payment/page.tsx`
- `apps/socrates/app/api/payment/create-order/route.ts`
- `apps/socrates/app/api/payment/callback/route.ts`
- `apps/socrates/app/(student)/payment/success/page.tsx`

### 当前问题

1. 国内用户更习惯验证码登录，不习惯为教育工具先设密码
2. 当前支付前必须先登录，转化链路偏长
3. “微信支付成功”和“微信身份登录成功”是两回事，现在没有真正打通
4. 登录页和注册页还没有“微信扫码快速进入”的承接能力

---

## 三、这次改造不应该做成什么

### 不建议直接做

- 先扫码付款，再无感自动登录

原因：

- 微信支付回调并不能直接给站点一个可登录用户身份
- 如果不先建立手机号或微信身份绑定，订单归属会变得不稳
- 一步把支付、身份、注册全部耦合，风险太高

### 不建议现在做

- 彻底替换掉所有密码登录逻辑

原因：

- 当前线上已有用户已在用密码模式
- 新方案应先增量兼容，再逐步迁移

---

## 四、推荐实施顺序

```text
Phase 1
手机号验证码登录 / 注册

Phase 2
微信扫码登录 / 账号绑定

Phase 3
支付成功后的补开户与自动承接
```

## 五、Phase 1: 手机号验证码登录 / 注册

### 目标

把“手机号验证码登录 / 注册”做成主入口，密码登录退居次入口。

### 用户路径

#### 登录页

用户进入 `/login` 后可选：

1. 手机号 + 验证码登录
2. 手机号 + 密码登录
3. 微信扫码登录（Phase 2 再开放）

#### 注册页

用户进入 `/register` 后：

1. 输入手机号
2. 获取验证码
3. 输入验证码
4. 补昵称、头像等最少信息
5. 首次登录后进入 `/select-profile`

### 页面改动

#### `/login`

新增模块：

- 手机号输入框
- 获取验证码按钮
- 验证码输入框
- 主按钮改为 `验证码登录`
- 保留 `密码登录` 切换入口
- 预留 `微信扫码登录` 区块

#### `/register`

新增模块：

- 手机号输入
- 获取验证码
- 验证码输入
- 首次注册信息补全

保留：

- 当前头像选择逻辑

可下沉为：

- 注册成功后的补资料页

### 服务端改动

建议新增接口：

- `/api/auth/send-code`
- `/api/auth/verify-code`
- `/api/auth/password-login`
- `/api/auth/password-register` 或保留现有 `/api/auth/register`

### 数据模型建议

建议新增一张验证码表：

```text
auth_verification_codes
- id
- phone
- purpose(login/register/bind_phone/reset_password)
- code_hash
- expires_at
- consumed_at
- send_count
- ip
- ua
- created_at
```

### 风控要求

- 同手机号限频
- 同 IP 限频
- 验证码短时失效
- 验证码只能消费一次
- 服务端只存 hash，不存明文

### 兼容要求

- 老用户仍可继续手机号 + 密码登录
- 新用户优先走验证码注册
- 不强制立即迁移老账号

### 复杂度评估

复杂度：中到大

原因：

- 会碰认证入口、短信服务、验证码风控和账号创建逻辑
- 但支付链路本身暂时不用大动

## 六、Phase 2: 微信扫码登录 / 绑定

### 目标

增加微信二维码快速登录能力，但优先做“绑定”，再做“直接新注册”。

### 为什么要先做绑定

因为当前系统已经有用户体系和手机号体系，先绑定最稳：

1. 已登录用户绑定微信
2. 未登录用户如果微信已绑定，直接登录
3. 未绑定时，再进入补手机号 / 建账号流程

### 用户路径

#### 场景 A：已登录用户绑定微信

1. 在设置页或登录页看到“绑定微信”
2. 扫码确认
3. 当前账号与微信 `openid/unionid` 绑定

#### 场景 B：未登录用户微信扫码

1. 打开登录页
2. 扫码
3. 如果已绑定账号，直接登录
4. 如果未绑定账号，进入补手机号流程

### 页面改动

建议新增：

- 登录页的“微信扫码登录”卡片
- 设置页的“微信绑定 / 解绑”区块

### 服务端改动

建议新增：

- `/api/auth/wechat/qr/start`
- `/api/auth/wechat/qr/poll`
- `/api/auth/wechat/bind`
- `/api/auth/wechat/unbind`
- `/api/auth/wechat/exchange`

### 数据模型建议

建议新增：

```text
user_auth_identities
- id
- user_id
- provider(wechat, phone)
- provider_user_id
- union_id
- open_id
- phone
- is_primary
- metadata
- created_at
- updated_at
```

至少要把“账号主键”和“外部身份”解耦，不要再只靠 `${phone}@student.local`。

### 风险

- 微信开放平台 / 服务号 / 公众号能力边界要先确认
- 微信扫码登录适用 Web 的具体接法要先定
- 不同微信体系拿到的是 `openid` 还是 `unionid`，会影响账户唯一性策略

### 复杂度评估

复杂度：大

原因：

- 这是外部身份系统接入，不是简单 UI 增加按钮
- 需要二维码会话、轮询、回调、安全态校验和账户绑定规则

## 七、Phase 3: 支付成功后的补开户与自动承接

### 目标

降低“先注册再支付”的阻力，但不直接把“支付成功”硬等同为“身份登录成功”。

### 推荐做法

不是做“支付后自动无感登录”，而是做：

`支付成功 -> 判断是否已登录 -> 未登录则补手机号验证码开户 -> 自动归属订单 -> 进入系统`

### 推荐路径

#### 已登录用户

1. 正常支付
2. 回调写订单和订阅
3. 成功页直接进入系统

#### 未登录用户

1. 支付成功页识别订单状态
2. 展示“输入手机号领取权益”
3. 用户通过验证码完成登录 / 注册
4. 系统把订单绑定到账户
5. 自动进入 `/select-profile` 或 `/dashboard`

### 页面改动

#### `/payment`

后续可支持“游客下单”模式，但第一期不建议先做。

#### `/payment/success`

新增状态分支：

- 已登录成功页
- 未登录补开户页
- 已有账号领权益页

### 服务端改动

建议新增：

- `/api/payment/claim-order`

职责：

- 根据支付成功订单和验证码登录身份，把待领取订单归属到当前账号

### 数据要求

如果要支持未登录先支付，订单表需要能在“未绑定用户”状态下先存在，再在后续补领。

建议为 `payment_orders` 增加：

```text
- pending_phone
- claimed_by_user_id
- claimed_at
- identity_snapshot
```

### 风险

- 订单被冒领
- 同手机号多账号冲突
- 已支付但未完成领单的异常状态
- 回调先到、用户后到时的状态一致性

### 复杂度评估

复杂度：大到很大

原因：

- 这是支付、认证、订单归属三条链路的交叉点
- 一旦设计不好，最容易出财务和账号归属问题

---

## 八、推荐的整体技术路线

### 第一阶段认证基线

短期建议继续保留现有 Supabase 账号体系，但逐步把“身份入口”和“内部 auth 实现”解耦：

```text
UI 层
  -> 手机号验证码登录
  -> 手机号密码登录
  -> 微信扫码登录

认证服务层
  -> 发送验证码
  -> 校验验证码
  -> 账户查找 / 创建
  -> 外部身份绑定

账号层
  -> profiles
  -> user_auth_identities
  -> supabase auth user
```

### 不建议长期继续依赖

```text
${phone}@student.local
```

它可以继续兼容旧用户，但不应继续作为长期唯一策略。

## 九、手机号验证码需要开通哪些服务

### 结论

要做手机号验证码登录 / 注册，除了前后端开发，还必须同步规划外部服务。

按当前架构，至少要准备 4 类能力：

1. 短信发送服务
2. 短信签名 / 模板 / 资质审核
3. 认证防刷能力
4. 密钥与运维配置

### 1. 短信发送服务

这是必须项，没有短信服务就无法真正上线验证码登录。

#### 方案判断

当前 `Supabase` 官方 Phone Auth 文档支持的短信提供商主要是：

- MessageBird
- Twilio
- Vonage
- TextLocal（社区支持）

这意味着如果继续完全依赖 Supabase Hosted Phone Auth 的默认供应商体系，对中国大陆场景并不理想。

因此，面向国内用户时更推荐：

- 方案 A：继续使用 Supabase 作为账号底层，但验证码短信走自建服务端发送
- 方案 B：后续如要彻底切原生 Phone OTP，再评估是否迁移或自托管

### 2. 国内短信资质、签名、模板

这是必须项，而且通常比开发更容易卡住排期。

#### 腾讯云短信

需要：

- 开通短信服务
- 创建国内短信实名资质
- 创建短信签名
- 创建验证码短信模板
- 审核通过后才能正式发送

#### 阿里云短信

需要：

- 开通短信服务
- 申请短信签名
- 申请验证码模板
- 等待审核通过

### 3. 验证码防刷与滥用防护

这是必须项，不能后置。

需要同步准备：

- 图形验证码或 Turnstile / hCaptcha
- 手机号发送频控
- IP 频控
- 单设备 / 单会话频控
- 验证码过期与单次消费

### 4. 密钥与运维配置

这是必须项。

至少要准备：

- 短信服务 Access Key / Secret
- 短信签名 ID / 模板 ID
- CAPTCHA Site Key / Secret
- 服务端环境变量管理
- 失败告警与发送日志记录

## 十、服务开通建议

### 推荐选择

如果主要在中国大陆使用，建议优先选：

1. 腾讯云短信
2. 阿里云短信

原因：

- 国内验证码场景成熟
- 审核、签名、模板、资质流程明确
- 更符合中国大陆的发送合规要求

### 推荐技术路径

不要第一期直接把验证码登录完全绑死在 Supabase Phone Auth 上。

更稳的做法是：

```text
前端
  -> 输入手机号
  -> 发送验证码

Socrates 服务端
  -> 调用国内短信服务商 API
  -> 保存验证码 hash 与风控记录
  -> 校验验证码
  -> 登录 / 注册 Supabase 用户
```

这样做的好处：

- 更适配国内短信服务商
- 风控更可控
- 后面接支付补开户、微信绑定时更灵活

### 推荐供应商选择顺序

#### 方案 1

腾讯云短信作为首选

适合原因：

- 国内短信文档和资质流程清楚
- 验证码场景成熟
- 对中国大陆 Web 产品常见

#### 方案 2

阿里云短信作为备选或双供应商兜底

适合原因：

- 也是成熟的国内短信供应商
- 后续如需要验证码兜底、双通道策略，有扩展空间

## 十一、服务开通排期建议

### 不能等开发结束后再申请

短信服务的资质、签名、模板审核很可能是整体项目的真实前置阻塞。

因此建议：

#### 本周立刻启动

1. 选定主短信供应商
2. 开通云账号与短信服务
3. 完成企业实名认证 / 资质报备
4. 提交短信签名
5. 提交验证码模板

#### 并行开发

1. 开发验证码表和接口
2. 开发登录 / 注册页新入口
3. 开发验证码发送与校验逻辑
4. 开发 CAPTCHA 和限流

### 推荐排期判断

- 技术开发可以并行推进
- 短信审核和资质申请不能拖到最后
- 如果本周不启动服务侧申请，Phase 1 很可能会被卡住

## 十二、验证码短信模板建议

### 模板必须表达清楚

建议模板要包含：

- 平台名称
- 验证码用途
- 失效时间

### 推荐模板方向

```text
【Socrates】您的验证码为 ${code}，用于登录或注册，5 分钟内有效，请勿泄露给他人。
```

### 注意

- 模板文案必须服从短信供应商审核规范
- 模板内容不要混入营销文案
- 平台签名要与业务主体、品牌名称一致

## 十五、建议实施顺序

### Step 1

先做认证基础重构设计：

- 新认证入口
- 验证码表
- `user_auth_identities` 表
- 登录 / 注册接口协议

### Step 2

落地手机号验证码登录 / 注册：

- 登录页
- 注册页
- 发送验证码
- 验证码登录 / 注册

### Step 3

补测试：

- 老密码登录回归
- 新验证码登录链路
- 手机号唯一性
- 风控限流

### Step 4

再做微信绑定和扫码登录

### Step 5

最后再做支付成功补开户和自动承接

---

## 十六、验收标准

### Phase 1 验收

1. 用户可用手机号验证码完成登录
2. 新用户可用手机号验证码完成注册
3. 老用户仍能用密码登录
4. 同手机号不会创建重复账号
5. 验证码具备基本限流和过期控制

### Phase 2 验收

1. 已登录用户可绑定微信
2. 已绑定微信用户可扫码登录
3. 未绑定微信用户会进入补手机号流程

### Phase 3 验收

1. 已登录支付用户成功后可直接进入系统
2. 未登录支付用户可在成功页补手机号领权益
3. 订单不会被错误归属或重复认领

---

## 十七、最终结论

当前最优路线不是一步做到“扫码付款后直接登录”，而是按下面顺序稳步推进：

1. 手机号验证码登录 / 注册
