# Socrates 多学科整合与导航重构规划

日期: 2026-03-16

## 1. 规划目标

本轮不是继续把平台功能平均铺开，而是先把 `Socrates` 重构成统一的学习主平台，围绕数学、语文、英语三科建立可持续扩展的信息架构。

核心目标有 4 个:

1. 把 `拍题/录题分析` 做成三科统一入口。
2. 把原 `Essay` 的核心能力归并到 `Socrates > 语文` 体系中。
3. 为英语补齐与数学、语文不同的专属模块，如听力、写作思路、作文批改。
4. 在不推倒现有结构的前提下，完成导航和数据模型的升级，支撑后续扩展到更多学科和更多工具模块。

一句话方向:

`Socrates` 负责统一学习平台，`Essay` 从独立产品逐步转为 `语文作文能力中心`。

---

## 2. 当前判断

结合现有代码和文档，当前结构已经具备部分多学科基础，但还没有真正完成“多学科 + 多模块”的产品组织。

当前已有基础:

- `workbench` 已支持学科切换思路，存在 `math / chinese / english` 的学科配置。
- OCR 与 prompt 层已出现 `writing`、`reading`、`listening` 等题型能力。
- `Essay` 已经形成较成熟的作文批改工作流，尤其在批注、移动端工作台、历史记录方面完成度较高。

当前主要问题:

- `Socrates` 顶部导航更适合“页面栏目”，不适合未来的“学科 + 模块”扩展。
- `workbench` 仍偏单页承载，后续继续叠加会越来越难维护。
- 数学、语文、英语虽然都能走“录题分析”，但结果页和工作流并不相同，不能机械共用一套页面。
- `Essay` 与 `Socrates` 仍是并列产品形态，长期会导致账户、订阅、历史记录、学习资产割裂。

结论:

- 方向不是单纯“加两个学科”。
- 而是要把平台从“单主工作台”升级为“统一学习平台 + 学科能力中心”。

---

## 3. 产品结构重组建议

### 3.1 顶层产品关系

建议把平台关系调整为:

```text
Socra Platform
├─ Landing
│  ├─ 品牌首页
│  ├─ 专题页 / SEO / GEO
│  └─ 转化入口
├─ Socrates
│  ├─ 统一学习主平台
│  ├─ 数学能力中心
│  ├─ 语文能力中心
│  └─ 英语能力中心
└─ Essay
   ├─ 短期保留独立入口
   ├─ 作为语文作文专题页/转化页
   └─ 底层能力逐步并回 Socrates
```

### 3.2 Essay 的归并策略

不建议立刻删除 `Essay` 独立站点，建议分两步:

第一步:

- 保留 `essay.socra.cn` 作为专题入口、SEO 入口、转化入口。
- 业务定义上改为 `Socrates > 语文 > 作文批改` 的外层入口。
- 账户、订阅、数据结构开始对齐。

第二步:

- 将作文历史、批注结果、学习记录、报告数据并入 `Socrates`。
- `Essay` 最终转为专题壳或轻独立工作台。

这样做的好处:

- 不打断现有流量和已完成的产品沉淀。
- 可以平滑迁移，不需要一次性重构全部页面。

## 4. 学科能力结构

建议按“学科 > 模块”方式组织，而不是按“全站功能页”堆叠。

### 4.1 数学

目标定位:

- 以题目分析和错题复盘为核心。

建议模块:

- 录题分析
- 几何图形识别
- 几何手工修正
- 辅助线添加
- 错题归档
- 变式练习
- 复习与巩固

说明:

- 数学链路相对单一，重点是稳定性、准确率和解题过程清晰度。
- 几何题必须保留“人工修正”能力，否则很难达到真实可用。

### 4.2 语文

目标定位:

- 建立“录题分析 + 阅读理解 + 作文能力”的组合型学科中心。

建议模块:

- 录题分析
- 阅读理解分析
- 基础知识题分析
- 古诗文/文言基础分析
- 作文批改
- 作文写作思路
- 后续可扩展: 素材积累、范文拆解、表达训练

说明:

- 语文不是一个单一工作流。
- `录题分析` 负责与数学统一入口。
- `作文批改` 和 `作文写作思路` 是独立价值模块，不能只作为题型分支处理。

### 4.3 英语

目标定位:

- 建立“录题分析 + 听力 + 写作”三条并行主能力。

建议模块:

- 录题分析
- 阅读理解与语法分析
- 完形/选择/改错分析
- 听力
- 英语写作思路
- 英语作文批改
- 后续可扩展: 口语、单词复习、句型训练

说明:

- 英语不能只按拍题逻辑设计。
- 听力和写作都属于独立工作流，需要单独模块。

## 5. 学习入口设计

### 5.1 统一入口原则

三科都保留统一学习入口:

- 拍题
- 上传图片
- 录入文本

统一入口负责:

- 用户进入成本低
- 学科识别或手动选科
- OCR 和结构化提取
- 分流到正确的学科模块

但统一入口之后，结果页不能完全统一。

### 5.2 统一入口后的分流逻辑

```text
统一学习入口
├─ 数学
│  ├─ 普通题分析
│  └─ 几何题分析/修正
├─ 语文
│  ├─ 阅读/基础题分析
│  └─ 写作相关入口
└─ 英语
   ├─ 阅读/语法分析
   ├─ 听力
   └─ 写作相关入口
```

### 5.3 关键设计原则

- 入口统一，降低学习成本。
- 结果分化，保证每个模块输出真正有用。
- 数据统一，确保都能沉淀到学习资产和复习系统。

## 6. 导航改造建议

## 6.1 不建议的方案

### 方案 A: 全部放顶部导航，下拉展开各科模块

问题:

- 一旦三科各自扩成多个模块，顶部导航会迅速膨胀。
- 桌面端尚可勉强使用，移动端非常不友好。
- 不利于后续增加物理、化学等学科。

### 方案 B: 全站直接改成重侧边栏

问题:

- 桌面端扩展性好，但移动端需要完全重做。
- 当前平台已有顶部导航和底部 tab 习惯，直接切重侧边栏会破坏已有使用路径。

## 6.2 建议采用的方案: 混合式导航

### 全局层

保留顶部导航，负责平台级入口:

- 学习
- 错题
- 复习
- 计划
- 报告
- 我的

### 学习层

进入“学习”后，再显示学科与模块导航:

桌面端:

- 左侧: 学科 + 模块树
- 内容区顶部: 当前模块二级切换

移动端:

- 底部 tab: 全局入口
- 页面顶部横向学科切换 chips
- 页面内部二级模块 tab 或 bottom sheet

### 推荐结构

```text
桌面端
┌ 顶部全局导航 ───────────────────────┐
│ 学习 | 错题 | 复习 | 计划 | 报告 | 我的 │
└────────────────────────────────────┘
┌ 左侧学习导航 ─────┬ 主内容区 ───────────┐
│ 数学              │ 当前模块页面         │
│ 语文              │                      │
│ 英语              │                      │
│                  │                      │
│ 录题分析          │                      │
│ 作文批改          │                      │
│ 写作思路          │                      │
└──────────────────┴──────────────────────┘
```

```text
移动端
顶部: 当前页面标题 + 学科切换
中部: 模块内容
底部: 学习 / 错题 / 复习 / 计划 / 我的
```

结论:

- 顶部导航继续保留。
- 学习域内部增加“学科/模块层”。
- 不做纯顶部下拉，也不做纯桌面侧栏照搬到手机。

## 7. 建议的路由结构

短期不必一次性推翻 `/workbench`，可以先保留统一入口页，再逐步拆分到更稳定的模块路由。

建议目标路由:

```text
/study
/study/math/problem
/study/math/geometry
/study/chinese/problem
/study/chinese/reading
/study/chinese/composition/review
/study/chinese/composition/idea
/study/english/problem
/study/english/listening
/study/english/writing/review
/study/english/writing/idea
```

短期过渡方式:

- `/workbench` 作为统一学习入口保留。
- 从 `workbench` 根据学科和模块跳转到对应路由。
- 待新结构稳定后，再弱化 `workbench` 的单页职责。

## 8. 架构改造建议

这次真正的核心不是 UI，而是底层抽象。

### 8.1 统一学习会话模型

建议建立统一的 `StudySession` 概念。

建议字段:

- `subject`
- `module`
- `input_type`
- `question_type`
- `source_app`
- `ocr_payload`
- `analysis_payload`
- `asset_refs`
- `review_status`
- `report_refs`

作用:

- 数学、语文、英语都能进入同一学习记录中心。
- `Essay` 迁移时不需要重新发明一套历史数据结构。

### 8.2 模块注册表

建议不要用大量硬编码 if/else 扩展学科和模块。

建议建立模块注册表:

```text
subject -> module -> input types -> prompts -> pipeline -> output cards
```

注册表负责描述:

- 这个模块是什么
- 支持哪些输入方式
- 调用哪套 prompt
- 输出哪种结果布局
- 是否进入错题/复习/报告

这样后面新增物理、化学时成本会明显降低。

### 8.3 学习资产中心

当前的“错题本”更偏数学语境，后面建议升级为更宽的学习资产中心。

建议包含:

- 错题
- 阅读题分析记录
- 作文批改记录
- 写作思路记录
- 听力练习记录

再在展示层保留“错题本”“作文记录”等子视图。

### 8.4 Essay 的技术合并方式

不建议一次性把整个 `apps/essay` 直接复制进 `apps/socrates`。

建议路径:

1. 先抽离共享数据结构和核心处理逻辑。
2. 再抽离批注工作台相关组件。
3. 最后决定是:
   - 继续保留独立 app 壳
   - 或完全并入 `Socrates`

这样风险最小。

## 9. 可行性与风险分析

## 9.1 可行性

整体判断: 可行，而且现在做比未来更便宜。

原因:

- 当前三科入口基础已存在。
- `Essay` 的能力已成熟，可复用而不是重造。
- 导航和模块层仍处于可调整阶段，用户心智尚未完全固化。

## 9.2 主要风险

### 风险 1: 把所有学科硬塞进同一个工作台

后果:

- 页面复杂度失控
- 状态难维护
- 未来每加一个模块都要改一堆分支

### 风险 2: 过度追求“完全统一”

后果:

- 数学、语文、英语工作流差异被抹平
- 看似统一，实际体验变差

### 风险 3: 一次性把 Essay 全量迁移

后果:

- 当前可用能力容易被迁移成本拖垮
- 影响已完成的作文批改链路

## 9.3 风险控制原则

- 统一入口，不强行统一所有页面。
- 统一数据底座，不强行一次性统一所有实现。
- 先产品归并，再技术收口。

## 10. 建议分阶段实施

### Phase 1: 信息架构与导航定型

目标:

- 明确 `Socrates` 作为统一学习平台的定位。
- 完成全局导航与学习域导航分层。

任务:

- 调整导航模型
- 明确三科模块树
- 设计新的学习首页/学习入口
- 保留 `workbench` 作为过渡页

完成标志:

- 用户能明确分辨“平台级栏目”和“学科级模块”
- 三科入口逻辑清晰，不再混在同一层导航中

### Phase 2: 语文作文能力接入

目标:

- 把 `Essay` 在业务上纳入 `Socrates > 语文`。

任务:

- 新增 `语文 > 作文批改`
- 新增 `语文 > 写作思路`
- 对齐账户、订阅、历史记录
- 设计从旧 `Essay` 到 `Socrates` 的跳转与映射

完成标志:

- 用户在 `Socrates` 内可直接进入作文能力
- 作文记录可进入统一学习资产体系

### Phase 3: 英语模块补齐

目标:

- 建立英语的差异化能力结构。

任务:

- 新增 `英语 > 录题分析`
- 新增 `英语 > 听力`
- 新增 `英语 > 写作思路`
- 新增 `英语 > 作文批改`

完成标志:

- 英语不再只是拍题页的一个学科标签
- 听力和写作成为独立模块

### Phase 4: 统一学习资产与复习系统

目标:

- 三科记录都能沉淀并用于复习和报告。

任务:

- 升级错题本为学习资产中心
- 按学科/模块分类沉淀记录
- 建立作文、阅读、听力、题目分析的统一复习机制

完成标志:

- 数学、语文、英语都能进入统一报告和复习系统

### Phase 5: Essay 的最终收口

目标:

- 明确 `Essay` 最终是专题壳还是完全合并。

任务:

- 评估 SEO、转化、产品独立性
- 评估是否保留单独域名入口
- 决定最终 app 架构

完成标志:

- 不再存在双份账户、双份学习记录、双份能力心智

---

## 11. 最终建议

当前最合理的长期路线不是:

- 继续把 `Socrates` 当数学主站
- 再把语文、英语零散补进去

而是:

1. 把 `Socrates` 升级为统一学习平台。
2. 把 `Essay` 业务上并入 `语文作文`。
3. 把英语补成完整学科，而不是拍题附属项。
4. 用混合式导航承接长期扩展。
5. 先统一数据和模块抽象，再逐步统一页面实现。

一句话总结:

`平台统一到 Socrates，学科按模块扩展，作文归入语文，导航改为全局顶栏 + 学习域内部导航，架构先抽象后迁移。`
