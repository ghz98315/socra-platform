# Socrates 账号模型下一阶段备忘

日期：2026-04-10
状态：下一阶段方案备忘
用途：记录账号体系从“双角色切换”向“单账号 + 家长二级密码 + 多学生隔离视图”的收口方向

## 1. 背景

当前 `socrates` 的账号心智仍然偏向：

- 一个 `profiles` 记录带 `role`
- 登录后可切到学生/家长入口
- 家长能力和学生能力在若干页面中直接按 `profile.role` 分流

这个结构在早期可用，但对真实用户会带来两个问题：

1. 用户会把“身份切换”误解成“账号切换”
2. 家长视图与学生视图虽然共用同一登录，但权限边界仍然太粗

## 2. 候选目标

下一阶段建议收口为：

- 一个用户只保留一个登录账号
- 默认进入学生主学习链路
- 需要进入家长看台时，走家长二级密码
- 如果未设置家长二级密码，则默认可进入
- 如果一个账号下绑定多个学生，所有学习数据必须按学生隔离展示，不能混在一起

简化后的产品心智应是：

`一个账号登录 -> 平时直接学 -> 需要看家长面板时再输家长密码 -> 多学生时先明确当前查看的是谁`

## 3. 为什么不并入当前 Phase 2

这不是纯前端切文案，而是账号与数据模型调整，至少会碰到下面几层：

- `profiles.role` 及相关类型定义
- 登录后默认跳转逻辑
- 家长页权限判断
- 导航中的角色展示与切换入口
- `family_groups` / `family_members` / `parent_students` 等关系表使用方式
- 以 `student_id`、`parent_id` 为条件的接口与报表

因此不建议把它混进当前 Phase 2 错题闭环收口里同时做。

## 4. 下一阶段建议拆分

### Phase A：产品与权限口径统一

- 取消“家长账号 / 孩子账号”对外表达，统一改为“同一账号下的学生视图 / 家长看台”
- 明确家长看台的触发入口、退出入口、忘记二级密码处理方式
- 明确“未设置二级密码默认可进入”的初始行为

### Phase B：数据模型收口

- 评估 `profiles.role` 是否降级为内部兼容字段，而不是主心智字段
- 增加家长二级密码字段或独立配置表
- 为“当前选中的学生”建立明确上下文，不再依赖角色切换暗含
- 多学生数据查询统一强制要求学生维度

### Phase C：页面与接口迁移

- 设置页新增家长二级密码设置与修改入口
- 家长面板入口改为密码门
- 家长相关 API 增加二级密码校验上下文
- 错题、复习、报告、任务等页面支持明确切换当前学生

## 5. 必须保持的约束

- 多学生数据不能混查
- 一个账号下可以看多个学生，但任何学习记录、错题、复习计划都必须带学生上下文
- 旧数据与旧页面在迁移阶段要可兼容，不能一次性打爆现有 `role` 依赖

## 6. 对当前阶段的影响

当前 Phase 2 / Phase 3 继续按现有模型推进，但要遵守两个原则：

- 新改动尽量少把“角色切换”写死在页面文案里
- 新链路优先用“当前学生 / 当前入口 / 下一步动作”来表达，而不是继续强化“你现在是家长还是学生”的账号心智
