# Socrates 数学错题深度闭环 V1 规划
日期: 2026-03-25

## 1. 结论

当前最重要的事情，不是继续横向扩模块，而是先把一个科目的关键学习闭环真正打穿。

建议优先级:

1. 先做 `数学错题深度闭环 V1`
2. 先把 `错题诊断 -> 苏格拉底式追问 -> 复习判定 -> 真会/假会 -> 关闭/重开 -> 家长协同` 跑通
3. 等这一条链路稳定后，再推广到语文、英语和其他 study 模块

原因:

- 数学错题是当前最接近产品核心价值的场景
- 现有 `error_sessions -> review_schedule -> reports -> family/parent` 链路已经有基础
- prompt 里已经出现了 `5 Whys 归因分析` 的雏形，但还没有真正进入结构化数据和产品闭环
- 现在系统更像“会分析、会记录、会安排复习”，还不是“能把不会真正变成会”

## 3. 当前关键缺口

### 3.1 归因没有产品化

当前“错因分析”更多停留在:

- prompt 文案
- 分析摘要
- 页面展示文案

但没有形成稳定的结构化字段，导致:

- 复习系统读不到“根因”
- 报告系统看不到“根因趋势”
- 家长侧无法知道“孩子到底卡在什么地方”

### 3.2 复习推进过粗

当前复习更多是:

- 到点提醒
- 完成一轮
- 推进下一阶段

但还没有严格区分:

- 独立做对
- 被提示后做对
- 做对但说不清原因
- 完全不会

这会导致“完成复习”不等于“真正掌握”。

### 3.3 真会 / 假会没有明确标准

现在缺少明确规则判断:

- 这道题到底是真会了
- 还是只是刚好做出来
- 是否应该关闭
- 是否应该继续复习
- 是否应该自动重开

### 3.4 家长看到的是汇总，不是问题画像

当前家长侧更偏:

- 学习时长
- 掌握数量
- 总体概览

但还看不到:

- 最近反复出现的根因
- 哪些是知识点问题，哪些是习惯问题
- 孩子当前最需要家长介入的风险点
- 家长应该怎么配合，而不是只看到结果

### 3.5 热力图逻辑还没有定义

“学习热力图”如果没有统一指标，会很快退化成装饰性图表。

它必须先定义:

- 横轴和纵轴是什么
- 热度分数怎么算
- 为什么热
- 热了以后系统和家长该做什么

## 4. V1 目标

本次 V1 不追求做大而全，目标是:

把“数学错题”做成一个真正能闭环的深度学习产品。

### 4.1 V1 核心链路

1. 学生上传错题
2. 系统输出结构化根因诊断
3. AI 进入苏格拉底式追问，不直接给答案
4. 学生完成自我解释与总结
5. 系统安排复习
6. 复习时先要求独立作答
7. 系统根据作答证据判断“真会 / 假会”
8. 结果进入报告、热力图和家长侧
9. 家长看到问题画像与协助建议

### 4.2 V1 不做的事

- 不先推广到所有科目
- 不先做复杂炫技图表
- 不先做大而全的统一 StudySession 重构
- 不先做所有模块统一接入

## 5. 产品定义

### 5.1 归因必须结构化

建议把错题归因拆成两层:

第一层: 表象问题层

这一层允许出现学生和家长最容易理解的表面描述，例如:

- 粗心
- 看错题
- 漏步骤
- 算错
- 时间不够
- 一紧张就乱

但这里必须明确:

- `粗心` 只能作为表象标签
- 不能直接作为最终根因
- 系统必须继续追问“为什么会粗心”

第二层: 根因类别

- 知识点缺失
- 概念混淆
- 审题偏差
- 计算失误
- 解题策略错误
- 学习习惯问题
- 心态/注意力问题

第三层: 5 Why 链路

建议结构:

- `surface_labels[]`
- `surface_error`
- `why_chain[]`
- `root_cause_category`
- `root_cause_statement`
- `evidence[]`
- `fix_actions[]`

其中:

- `surface_labels[]` 是学生或系统首先观察到的表象标签，例如“粗心”“漏看条件”
- `surface_error` 是表层错误
- `why_chain[]` 是逐层追问链
- `root_cause_category` 是根因大类
- `root_cause_statement` 是一句明确的根因定义
- `evidence[]` 是为什么做出这个判断
- `fix_actions[]` 是后续修正动作

建议额外增加一个字段:

- `root_cause_depth`

含义:

- `1` = 只停留在表象
- `2` = 已追到行为习惯层
- `3` = 已追到稳定模式层，可以指导后续干预

V1 的最低要求:

- 归因不能只停在 `root_cause_depth = 1`
- 如果系统最终还在“粗心”，则视为归因失败，需要继续追问

### 5.2 苏格拉底式追问必须有边界

不是“多问几个问题”就叫苏格拉底式提问。

建议规则:

1. 一次只问一步
2. 先问学生“你当时是怎么想的”
3. 再问“哪一步其实没有依据”
4. 再问“如果重做，你先检查什么”
5. 最后要求学生自己总结

并且必须覆盖下面这个关键追问:

- 如果学生说“我就是粗心”
- 系统必须继续往下挖
- 直到找到:
  - 是审题习惯问题
  - 还是草稿管理问题
  - 还是计算检查习惯问题
  - 还是时间分配问题
  - 还是情绪/专注问题
  - 还是知识点不熟导致假性粗心

建议加入一条硬规则:

- 不允许用“粗心”结束本轮根因归因
- 除非系统已经补出“粗心背后的行为模式”

禁止:

- 一上来直接告诉答案
- 连问多步导致学生只是在跟读
- 只做鼓励，不推动学生形成判断标准

### 5.3 真会 / 假会判定

建议不要采用“独立一次做对就直接关闭”。

更稳的规则:

- `真会候选`
  - 本轮复习独立做对
  - 未向 AI 求助
  - 能复述核心思路
- `真会`
  - 至少连续两次间隔复习独立做对
  - 且都不依赖 AI 提示
- `假会`
  - 需要 AI 提示后才做对
  - 或做对但说不清为什么
  - 或变式题不会
- `关闭`
  - 达到“真会”标准后进入关闭状态
- `重开`
  - 同类题再次出错
  - 或家长/系统发现掌握并不稳定

### 5.4 复习证据必须落库

每一次复习不能只记录“完成了没”，建议记录:

- 是否先独立作答
- 是否求助 AI
- AI 提示次数
- 是否最终做对
- 是否能复述方法
- 作答耗时
- 自信度
- 是否通过变式题确认

## 7. 阶段计划

## Phase A: 核心模型与规则落地

目标:

- 明确什么叫根因
- 明确什么叫真会/假会
- 让系统先有统一语言

输出:

1. 根因 taxonomy 文档
2. 5 Why 结构化输出定义
3. 复习证据结构定义
4. 掌握状态机定义
5. 真会 / 假会判定规则

验收:

- 团队内部对“真会”“假会”“关闭”“重开”没有歧义
- 数据结构足够支撑后续页面、报告、家长侧

## Phase B: 数学错题诊断页与苏格拉底追问

目标:

- 把“错因分析”从文案升级为产品流程

输出:

1. 数学错题诊断结果页
2. 根因卡片
3. 5 Why 链路展示
4. 修正动作 checklist
5. 学生自我总结输入区
6. 苏格拉底追问流程
7. “粗心不是根因”的继续追问机制

页面上至少要显示:

- 我错在什么地方
- 为什么会错
- 根本原因是什么
- 下次先检查什么
- 这一题对应哪个知识点
- 哪些行为习惯正在反复制造这个问题

验收:

- 学生完成后，不只是“看懂答案”
- 而是系统能拿到结构化根因和学生自我总结

## Phase C: 复习判定与艾宾浩斯闭环

目标:

- 让复习真正和掌握状态绑定

输出:

1. 复习尝试记录
2. 独立作答优先流程
3. AI 介入受控流程
4. 真会 / 假会判定逻辑
5. 关闭 / 重开逻辑
6. 艾宾浩斯间隔与证据质量联动

建议规则:

- 独立做对但耗时过长，不直接算稳定掌握
- 求助后做对，进入更短复习周期
- 连续两次间隔复习独立做对，才可关闭
- 原题会但变式题不会，不关闭
- 同类题再次出错，自动重开

验收:

- 系统不会再把“做完一轮”误判成“已经会了”

## Phase D: 家长侧画像、热力图与提醒

目标:

- 把学习数据翻译成家长可行动的信息

输出:

1. 家长问题画像卡
2. 根因趋势
3. 真会 / 假会占比
4. 高风险对话提醒
5. 家长协助建议
6. 学习热力图

### 热力图建议先做什么

建议第一版先做以下两类热力图中的一种:

1. `知识点 x 根因类别`
2. `日期 x 根因类别`

热度分数建议由以下因素组成:

- 复发次数
- 最近 7 天出现频率
- 是否仍未关闭
- 是否逾期复习
- 是否被判定为假会

热力图的重点不是“热”，而是“热了以后怎么处理”。

### 家长端建议展示的重点

不要先给家长看大量原始消息，而是先给:

1. 当前最主要的 3 个问题
2. 最近反复出错的知识点
3. 哪些问题属于知识点，哪些属于习惯
4. 哪些题是“会做但不稳定”
5. 家长下一步应该怎么配合

### 不合适对话提醒

建议先抓 4 类:

- 明显摆烂
- 作弊倾向
- 强烈挫败
- 自我否定

推送给家长时不要只转发原话，应该附:

- 风险标签
- 触发原因
- 建议回应方式

## 8. 推荐的开发顺序

建议按下面顺序执行:

1. 数学错题根因结构化模型
2. 复习尝试记录与真会/假会判定
3. 学生端“根因诊断 + 苏格拉底追问 + 自我总结”流程页
4. 复习调度与关闭/重开逻辑
5. 家长端问题画像与干预建议
6. 热力图
7. 再抽象到其他科目

## 9. 第一阶段建议的实际交付范围

为了避免一次铺太大，建议先做一个非常明确的里程碑:

### 里程碑名称

`数学错题深度闭环 V1`

### 里程碑范围

包含:

- 数学错题结构化归因
- 5 Why 根因链路
- 苏格拉底式追问
- 学生自我总结
- 复习尝试记录
- 真会 / 假会判定
- 关闭 / 重开
- 家长问题画像基础版

不包含:

- 全科推广
- 复杂 BI 级报表
- 大一统 session 重构
- 大规模视觉优化

## 10. 验收标准

如果 V1 做完，至少应该满足下面这些标准:

1. 一道数学错题进入系统后，系统能给出结构化根因，而不是一段泛泛文案
2. 学生在 AI 引导下，能明确说出“我错在哪、为什么错、以后怎么防”
3. 每轮复习都能记录真实证据，而不是只记录“完成了”
4. 系统能区分“真会候选”“真会”“假会”“未掌握”
5. 复习调度会根据掌握质量变化，而不是固定死规则
6. 家长可以看到孩子当前最主要的问题画像
7. 家长能看到可执行协助建议，而不是只有统计数字

---

## 11. 我建议你本轮先确认的事项

在正式进入开发前，建议先确认下面 5 个决策:

1. 是否同意“先数学，后扩科目”
2. 是否同意“独立一次做对，只算真会候选，不直接关闭”
3. 是否同意新增独立表保存归因与复习尝试，而不是继续堆进 payload
4. 是否同意家长侧先看“问题画像与建议”，而不是先做复杂大屏
5. 是否同意第一版热力图先做“知识点 x 根因类别”或“日期 x 根因类别”，不做炫图
